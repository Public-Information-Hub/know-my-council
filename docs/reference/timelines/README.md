# Timeline Notes

Timeline notes capture short, high-signal project context over time.

They exist to help future maintainers and reviewers understand:

- why the repository looks the way it does
- what major direction-setting work happened in early phases
- which decisions were made when the project was still forming

Timeline notes are not:

- a changelog (use `CHANGELOG.md` for releases)
- meeting minutes
- a substitute for issues/PRs

## When to add a timeline note

Add a timeline note when there is a meaningful “phase shift” such as:

- a major re-organisation of docs or repository structure
- the introduction of a new platform layer (ingestion, read models, search indexing)
- a significant governance or community policy change

Keep notes concise and factual. Prefer linking to ADRs when decisions are involved.
